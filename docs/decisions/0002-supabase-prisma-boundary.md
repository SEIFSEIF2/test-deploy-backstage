# 0002 — Supabase Auth ↔ Prisma boundary

## Context

Slice 1 uses Supabase as both the auth provider and the Postgres database, and Prisma as the ORM. Supabase ships an `auth` schema containing `auth.users`, `auth.sessions`, and friends — it manages those tables on its own and migrates them when it ships features. Prisma, given the chance, will introspect them and try to manage them too. That way lies a messy migration the first time Supabase changes its auth schema underneath us.

The slice-1 plan calls this out as the one sharp edge to get right on day one.

## Decision

The `auth` schema is **owned by Supabase, end to end**. Prisma never models, migrates, introspects, or references the `auth.*` tables. Everything else in the `public` schema is owned by Prisma — Prisma migrations are the source of truth there.

Concretely:

1. **Prisma schema** declares `schemas = ["public"]` (not `auth`) in the datasource block. `multiSchema` stays off unless a future slice needs a non-auth secondary schema.
2. **`crew_member.id`** is the same UUID as the auth user's `auth.users.id`. The link is by value, not by Prisma-modeled FK. In Postgres there is a real FK constraint added by hand in a SQL migration, *not* generated by Prisma:
   ```sql
   alter table public.crew_member
     add constraint crew_member_auth_user_fk
     foreign key (id) references auth.users(id) on delete cascade;
   ```
   This SQL lives in a `prisma/migrations/.../migration.sql` file written manually after `prisma migrate dev`, so the FK is checked into source control without Prisma trying to own the target.
3. **Sign-up flow** creates the `auth.users` row first (via `supabase.auth.signUp`), then inserts the matching `crew_member` row using the returned user id. A Postgres trigger could automate this, but slice 1 keeps it in application code where it's debuggable.
4. **No Prisma model for sessions, identities, or auth refresh state.** Reading the current user goes through `@supabase/ssr` server helpers, never through Prisma.

## Consequences

- The Prisma `crew_member` model has `id String @id @db.Uuid` with **no `@default(uuid())`**. The id comes from Supabase, not from Prisma's default.
- A new test member can only be seeded by inserting into `auth.users` via the Supabase admin API (or the SQL editor with `auth.admin_create_user`) and then running the Prisma seed for the matching `crew_member` row. Seed scripts must do both halves.
- Deleting a Supabase user cascades into `crew_member` because of the hand-written FK. Deleting a `crew_member` row alone leaves a dangling auth user — sign-out and delete-from-Supabase must happen in that order in application code.
- If Supabase ships an auth schema change, our Prisma migrations are unaffected. We pick up the change automatically.

## How to apply when adding a new model

If a new table needs to point at a person, point it at `crew_member.id`, not `auth.users.id`. The boundary lets the rest of the app stay Prisma-only.
