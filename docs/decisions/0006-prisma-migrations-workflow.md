# 0006 ‚Äî Prisma migrations workflow (with the auth FK appended)

## Context

The Prisma schema is written but no migrations have been applied yet. Two things need to happen the first time we run a migration:

1. Prisma's postinstall scripts (`@prisma/engines`, `prisma`) were skipped by pnpm 10 ‚Äî they need approval before `prisma generate` or `prisma migrate dev` will work.
2. The `crew_member.id` ‚Üî `auth.users.id` foreign key from [0002](0002-supabase-prisma-boundary.md) cannot be generated by Prisma (the `auth` schema is owned by Supabase). It has to be appended by hand to the generated init migration's `migration.sql`.

We need a written workflow so future-me (or a different agent) doesn't blow past either of those.

## Decision

The init migration is created in three steps. The first run is the only one with manual stitching; every subsequent migration is the standard Prisma flow.

### Step 1 ‚Äî One-time setup

```bash
# Approve Prisma's postinstall scripts (pnpm 10 requires this once).
pnpm approve-builds
```

Approve `@prisma/engines` and `prisma`. This writes the approval into `package.json` under `pnpm.onlyBuiltDependencies` (or similar). Commit that change.

### Step 2 ‚Äî Configure environment

In `.env.local` (gitignored), set:

```
DATABASE_URL=postgresql://postgres.<ref>:<password>@aws-N-<region>.pooler.supabase.com:5432/postgres
```

URL comes from Supabase Dashboard ‚Üí Settings ‚Üí Database ‚Üí **"Session pooler"** tab (not Transaction pooler ‚Äî that strips advisory locks and breaks migrate; see [0007](0007-prisma-7-adapter-pattern.md)). Prisma 7 only takes one URL. The migrations workflow reads it from `prisma.config.ts` at the repo root; the runtime client reads it via the `pg` adapter in `lib/prisma.ts`.

Prisma loads `.env` automatically but **not** `.env.local`. Source it manually before running migrate commands:

```bash
set -a && . ./.env.local && set +a
pnpm prisma migrate dev --name <change>
```

Or symlink `.env.local` ‚Üí `.env` if you want Prisma to pick it up directly.

### Step 3 ‚Äî Apply the init migration

```bash
pnpm prisma migrate dev --name init
```

This generates and applies the migration in one step. Prisma's `_prisma_migrations` history table is created in `public` if not already there.

> **Don't include the auth FK in the Prisma migration.** The first draft of this decision said to append `prisma/sql/crew_member_auth_fk.sql` to the generated `migration.sql`. That fails: Prisma validates every migration against a temporary shadow database that is a *clean* Postgres without Supabase's `auth` schema, so any statement referencing `auth.users` throws `schema "auth" does not exist`. The FK is applied separately in Step 4.

### Step 4 ‚Äî Apply the auth FK via the Supabase MCP

The `crew_member.id` ‚Üí `auth.users.id` foreign key is intentionally outside Prisma's migration history (boundary rule from [0002](0002-supabase-prisma-boundary.md)). Apply it via the Supabase MCP's `execute_sql` (or `psql`) with the contents of `prisma/sql/crew_member_auth_fk.sql`:

```sql
alter table public.crew_member
  add constraint crew_member_auth_user_fk
  foreign key (id) references auth.users(id) on delete cascade;
```

The snippet is idempotent in spirit but not literally (Postgres has no `ADD CONSTRAINT IF NOT EXISTS`). If the FK already exists you'll get an error ‚Äî that's the signal it's already applied.

If we ever `prisma migrate reset`, re-apply this snippet afterward.

### Every migration after init

Run `pnpm prisma migrate dev --name <change>` normally. The auth FK is independent of Prisma's history, so future migrations don't need any stitching.

## Consequences

- The first migration has one hand-edit step that's visible in the diff (the appended FK SQL). Code review can confirm it.
- `pnpm prisma generate` (which only produces types, no DB touch) should be run any time `schema.prisma` changes so `lib/prisma.ts` typechecks. CI can run it automatically; locally we run it on demand.
- A new dev environment runs `pnpm install` ‚Üí `pnpm prisma generate` ‚Üí `pnpm prisma migrate deploy` to catch up.
- If we ever regenerate the init migration from scratch (`prisma migrate reset` then `migrate dev`), we have to re-append the FK snippet. `prisma/sql/crew_member_auth_fk.sql` exists exactly so that paste is a one-liner.
