# Crew OS — design.md

A single design brief for the whole product. Hand this file to Claude as the
source of truth for designing the interface. It covers what the product is,
who uses it, the design language, every page, the shared components, the
states to handle, and a kickoff prompt at the end.

> Status: design phase. No code yet. The goal of this file is screens, not a build.

## 0. How to use this file

This file is the brief. When designing:

1. Read sections 1 to 3 first — they set the product and the design language.
2. Design page by page from section 6, in the build-order given.
3. Every page must obey the design language in section 3 and reuse the shared
   components in section 5. Do not invent new patterns per page.
4. When a page needs a decision this file does not answer, make the call that
   is simplest for an inexperienced first-time user, and note the assumption.

The north star: a person who has never seen this tool should understand their
screen in five seconds, with no training.

## 1. What Crew OS is

Crew OS is an internal operations platform for a small creative company. It
replaces a scatter of WhatsApp chats, Google Drive folders, and half-used
tools like Jira and Notion with one place.

It does five things:

1. Plans the crew — who holds which role, who is free, who is overloaded.
2. Tracks the work — projects, tasks, and clean handoffs between people.
3. Onboards new people — every newcomer has a guided start and a mentor.
4. Holds the knowledge — SOPs, role guides, and policies, in one searchable base.
5. Builds the culture — a shared space for ideas, plus recognition.

It is used by one company at a time, under 50 people. It is built to be
reused for another company later, but it is not a multi-tenant SaaS.

## 2. Who uses it, and what they need

The company runs on a rotating mix of founders, employees, and apprentices
(short-term interns, often students). Skill level is low and turnover is high.
That single fact drives every design decision.

Three kinds of user:

- **Apprentice / employee** — the everyday user. New, often unsure, here for
  three months or so. Needs an obvious "what do I do now" and zero ambiguity.
  Most screens are designed for this person first.
- **Lead** — runs a team or a project. Needs to see status, assign work, and
  spot trouble early.
- **Admin / founder** — runs the company. Needs the capacity picture, the
  roadmap, and oversight.

Design implication: clarity beats density. A confused apprentice is the main
failure mode. When in doubt, show less and label it well.

## 3. Design language

The product is for a creative entertainment brand, so the interface should
feel premium and considered, never like a generic admin panel. But it is a
work tool used all day, so it must stay calm and legible. Premium, not loud.

### 3.1 Principles

- **Calm by default, color on purpose.** Surfaces are near-black and grey.
  The accent color appears only where it carries meaning — attention, identity,
  status. If the accent is everywhere, it means nothing.
- **One pattern per job.** A status is always shown the same way. A person is
  always shown the same way. Consistency is the product feeling trustworthy.
- **Five-second test.** Every screen must orient a first-time user in five
  seconds. If it cannot, simplify the screen, do not add a tooltip.
- **Read first, edit deliberately.** Dashboards show and link out. Editing
  happens on focused screens, not everywhere at once.
- **Flat and clean.** No gradients, no heavy shadows, no decorative noise.
  Thin borders, generous whitespace, flat surfaces.

### 3.2 Theme — SKAM dark

The default theme is dark.

- Page background:        `#0E0E10`
- Card background:        `#161618`
- Raised / inset card:    `#1A1A1C`
- Border:                 `#2A2A2E`  (used at 0.5px)
- Divider line:           `#232327`
- Text primary:           `#F2F2F0`
- Text secondary:         `#A8A8AE`
- Text muted:             `#8A8A90`
- Text dim:               `#5C5C62`
- Accent — SKAM red:      `#E24B4A`
- Success / healthy:      `#5DCAA5`
- Warning / at risk:      `#EF9F27`
- Info / neutral active:  `#85B7EB`
- Danger / critical:      `#E24B4A`  (shares the accent red)

A light theme may come later. Design dark first; keep contrast high enough
that a light theme is a re-skin, not a redesign.

### 3.3 Type

- One sans-serif family throughout. Sentence case everywhere — never Title
  Case, never all-caps labels.
- Rough scale: page title 22px, section title 16px, body 14px, meta 12px.
- Two weights only: regular and medium. Medium for titles and labels. No bold
  mid-sentence, no heavy weights.

### 3.4 Shape and spacing

- Card corner radius: 12px. Inner element radius: 8px. Pills: fully rounded.
- Borders: 0.5px, the border color above. Borders are quiet, not structural.
- Whitespace does the separating. Group with space before reaching for a line.
- Vertical rhythm in steps of 8px.

### 3.5 Accent discipline

Red (`#E24B4A`) is allowed only for: the person's own identity marker, an
overdue or critical state, onboarding progress, and primary actions. Never as
a default fill or decoration. Status colors (success, warning, info) are for
status only. A screen with one red thing on it is working correctly.

## 4. Information architecture

Top-level navigation, in order:

1. **Cockpit** — the person's own home.
2. **Projects** — the project workspaces: board and timeline per project.
3. **Crew** — people: the directory, profiles, and the admin Crew Board.
4. **Knowledge** — the knowledge base and doc search.
5. **Vault** — the shared culture space.

A persistent left sidebar holds this nav, the company switcher (one company
for now, but the slot exists), and the current user. A top bar holds global
search and the current project context.

The hierarchy of data: Company contains Projects; Projects contain Tasks,
Timelines, and role demand. People belong to the Company, never to a project.
A person is linked to a project through an allocation. One special project,
"Operations", is the standing home for company work that is not a real
project — onboarding, recruiting, the Vault.

## 5. Shared components

Design these once. Every page reuses them exactly.

- **Person chip** — avatar (initials) + name. The only way a person is shown.
- **Status pill** — small rounded label. Fixed color per status:
  backlog/unscoped grey, to do grey, in progress info-blue, in review warning,
  done success, canceled muted.
- **Project tag** — small neutral tag naming a project. Every task and doc
  carries one.
- **Capacity light** — a colored dot or pill for role health: critical red,
  under capacity warning, staffed success, surplus info.
- **Metric card** — a muted label above, a large number below. For summary
  stats. No border, inset background.
- **Card** — the default container: card background, 0.5px border, 12px radius.
- **Empty state** — a calm one-line message plus, where useful, one action.
  Never a broken-looking blank panel.
- **Progress bar** — thin, one or more colored segments. For onboarding and
  allocation.
- **Section header** — a 16px medium title, optional meta note on the right.

## 6. The pages

Design in this order. Each page lists its purpose, who sees it, layout,
the key elements, and the states it must handle.

### 6.1 Crew Cockpit  — build first

**Purpose.** The first screen on login. Answers: what do I do today, what is
blocked, where am I in my time here. A read-mostly dashboard that links out
to the real editors.

**Who.** Any logged-in person, showing only their own data.

**Layout.** Single column, centered, about 980px max width. Six blocks top
to bottom, ordered by urgency:

1. Header strip — person chip, and "Day 6 of 90 · ends 20 Aug" on the right.
2. Onboarding tracker — full width. Only shown while the person is onboarding.
3. My tasks — left column, wider — with a right rail beside it.
4. Right rail — My allocation, then Handoffs, stacked.
5. My roadmap — full width.
6. Spotlight — full width, at the bottom. The company recognition block.

**My tasks.** A list of the person's open tasks. Each row: status pill, title,
a sub-line "project · solo or with [name]", and a due date that turns red when
overdue. A "View board" button links out.

**My allocation.** One total percent across all projects, a split bar by
project, and per-project rows. If the total exceeds 100, it turns red with an
"overloaded" note — the early-warning signal.

**Handoffs.** Two short lists: "to fill" and "received". A to-fill item that
blocks a task shows a "blocks Done" note.

**My roadmap.** The person's own milestones for the next six weeks as simple
time bars. Header says "my milestones only" — they see their slice, not the
company roadmap.

**Spotlight.** The recognition block, shown to everyone. It highlights the
currently spotlighted person or people — their person chip, the period, and a
short free-text reason written by an admin. If the viewer is the spotlighted
person, the block gives them a warm, clear "you are spotlighted" treatment.
This is the same Spotlight defined in full in section 6.9; the Cockpit is its
main home because recognition only works if people actually see it.

**States.** Onboarding (tracker shown, few tasks) · active (tracker hidden,
tasks full) · overloaded (allocation red) · empty (calm message, no broken
panel) · wrapping up (a gentle "finish your handoffs" nudge near the end date) ·
no current spotlight (the Spotlight block shows a calm "no one spotlighted
right now" line, never a broken panel).

### 6.2 Project workspace

**Purpose.** Where the work of one project lives — its board and its timeline.

**Who.** Anyone allocated to the project. Leads can edit; others mostly view.

**Layout.** A project header (name, the roles it needs, key dates) with a view
switcher: Board and Timeline.

**Board.** Columns by status: Backlog, Unscoped, To do, In progress, In review,
Done. Cards show title, project-relative id, assignee chip, discipline labels,
due date, and a small marker if the task has dependencies. The Unscoped column
is deliberate — it holds work not yet defined enough to start.

**Timeline.** Milestones and tasks across time. Each milestone carries a
visibility marker: project-wide, leads-only, or restricted. Restricted items
appear only to permitted viewers — design both the full view and the
reduced view.

**Handoff.** Opening a task shows its handoff doc — six fixed fields (what it
is, current status, done so far, still left, where the files are, gotchas, who
to ask). A task cannot move to Done until the handoff is complete; show this
as a clear gate, not a silent failure.

**States.** Empty project (no tasks) · a task blocked by an unfilled handoff ·
a restricted timeline seen by a non-permitted user.

### 6.3 Crew Board  — admin

**Purpose.** The staffing picture. Shows every role, whether it is under or
over capacity, and suggests moving surplus people into shortages.

**Who.** Admins and leads.

**Layout.** A company switcher and a month selector at the top. Summary metric
cards: total crew, roles staffed, roles needing hiring, surplus roles. Then
roles grouped by family (Production, Post, Creative, Game, Tech, Growth,
Operations).

**Capacity by role.** One row per role: the role name, a bar where a grey band
is the target range and the colored fill is the actual headcount, the count
versus the range, and a capacity light. Empty band is a hole; fill past the
band is surplus.

**Redeployment.** A list of suggested moves: a surplus person, an arrow, a
shortage role, and the reason (a matching secondary skill or a stated learning
goal). Each has a "plan move" action.

**Forecast.** A grid: roles down the side, the next four to six months across.
Each cell is a projected headcount, colored by health. This shows shortages
before they happen, because people's end dates are known.

**States.** A role at zero (critical) · a role in surplus · a forecast cliff
(a future month dropping below target) called out with a clear warning.

### 6.4 Crew profile

**Purpose.** One person's full record. The data that feeds capacity, matching,
and the exit summary.

**Who.** The person themselves, plus leads and admins.

**Layout.** A header with the person chip, role, contract type, and lifecycle
status. Then sections: primary role and secondary skills (each skill with a
level — the skill tree); availability (start and end dates, work mode, weekly
capacity); current allocations across projects; a contribution log of
completed work; and any redeployment history.

**States.** Incoming (not started yet) · active · wrapping up (the contribution
log is near-final and an exit summary can be generated) · alumni.

### 6.5 Onboarding tracker  — admin and mentor view

**Purpose.** Make sure every newcomer has an assigned mentor and a guided
start. The company-wide view of who is being onboarded.

**Who.** Admins, and mentors for their assigned people.

**Layout.** A list of people currently onboarding. Each entry: person chip,
role, start date, mentor, and a progress bar of onboarding steps. Opening one
shows the step checklist, the agreements-to-sign gate, and the role onboarding
pack. A person cannot be marked fully started until required agreements are
signed.

**States.** No one onboarding (calm empty state) · a newcomer with no mentor
assigned (flag this — it should not happen) · agreements unsigned (the start
is gated).

### 6.6 Knowledge base

**Purpose.** The single source of truth for SOPs, role guides, and policies.
It is also what the AI assistant reads, so trust matters.

**Who.** Everyone reads. Owners maintain their docs.

**Layout.** A searchable, filterable list of docs. Each doc shows its title,
owner, project or company scope, and a verification status: verified, draft,
verification expired, outdated, empty. A management view lets owners see
everything that is stale or unowned at a glance.

**Doc import.** A clear flow to bring in Markdown and Google Docs files, so a
scattered set of docs can be consolidated fast.

**Knowledge gaps.** A list of questions the AI could not answer. Each one is a
doc waiting to be written.

**States.** Empty base (a strong "import or create your first doc" state) ·
a doc with expired verification · an unowned doc.

### 6.7 Vault

**Purpose.** The shared culture space — a place for everyone to post ideas,
quotes, inspiration, sources, and projects, with light voting and comments.

**Who.** Everyone.

**Layout.** A feed of posts grouped or filterable by kind (idea, quote,
inspiration, source, project). Each post: author chip, kind tag, title, body
or link, vote count, comment count. A contributor highlight area can show the
most active posters — this is about Vault activity, and is separate from the
company Spotlight in section 6.9.

**States.** Empty feed (an inviting "share the first thing" state) · a post
with no votes or comments yet.

### 6.8 AI assistant  — design last

**Purpose.** A role-aware assistant that answers questions from the knowledge
base, in the person's own language, and suggests skills to learn.

**Who.** Everyone, scoped to what they can see.

**Layout.** A simple ask-and-answer panel. An answer always shows its sources
from the knowledge base. A question it cannot answer is captured as a
knowledge gap.

**Design caution.** Do not make the assistant feel authoritative beyond its
source. Always show sources. An answer with no source should look visibly less
certain than one with sources.

### 6.9 Spotlight  — recognition

**Purpose.** Recognize people in front of the whole company. A small, warm,
low-effort way to make good work visible. This matters more than it looks: in
a company that pays partly in experience, being seen is part of the pay.

**Who sees it.** Everyone. Its main home is a block at the bottom of every
person's Cockpit (section 6.1), so it is seen daily without anyone navigating
to it. A simple history of past spotlights also lives under Crew.

**Who runs it.** A lead nominates a person, and an admin confirms. Two
lightweight steps: nomination keeps it from being one manager's pet pick;
confirmation keeps it intentional. Design both the lead's "nominate" action
and the admin's small confirm view.

**What a spotlight is.** One or more people, a period (for example a month),
and a short free-text reason written by the admin. The reason is flexible on
purpose — it can celebrate a shipped project, a quiet helper, or steady
growth, not just whoever closed the most tasks.

**The Cockpit block.** Shows the current spotlight: the person chip or chips,
the period, and the reason. When the viewer is the spotlighted person, give
them a distinct, warm treatment — they should feel it, not just read it. Keep
it tasteful and on-theme; the accent red may be used here because recognition
is a moment that earns attention.

**States.** No current spotlight (a calm "no one spotlighted right now" line,
never a broken panel) · viewer is the spotlighted person (warm treatment) ·
a pending nomination waiting on admin confirmation (visible only to admins).

## 7. Cross-cutting states

Every page must handle these, not just the happy path:

- **Empty** — calm one-line message, optional single action. Never a blank panel.
- **Loading** — a quiet skeleton, no spinner theatre.
- **Error** — plain language, a way to retry. Never a raw error string.
- **Permission** — when a person cannot see something, show a clean "you do not
  have access" rather than a broken layout. This matters most on restricted
  timelines.
- **First run** — a brand-new company has almost no data. Every page needs a
  first-run state that guides the first action.

## 8. Accessibility and quality bar

- Text on any background must stay clearly legible. Test every color pair.
- Never rely on color alone — pair a status color with a word or an icon.
- Tap and click targets comfortably large; this is used quickly.
- Sentence case, plain language, no internal jargon in the interface.
- The whole thing should feel like one product — same spacing, same components,
  same voice on every screen.

## 9. Out of scope

Do not design these now:

- A multi-tenant sign-up or billing flow.
- Real-time collaborative editing.
- An in-app chat or messaging system.
- A native mobile app — responsive web is enough.
- A heavy notifications center — a weekly digest covers it.
- Anything public-facing.

## 10. Kickoff prompt

Copy the text below to start the design work.

---

You are designing Crew OS, an internal operations platform for a small
creative company. The full brief is the design.md you have been given — read
it before anything else, especially sections 1 to 3 for the product and the
design language.

Design the screens for Crew OS, page by page, in the order listed in section 6,
starting with the Crew Cockpit. For each page:

1. Produce a high-fidelity screen design in the SKAM dark theme defined in
   section 3.2, obeying the design principles in section 3.1.
2. Use only the shared components defined in section 5. Do not invent a new
   pattern when one already exists.
3. Design the real states the page must handle, listed per page in section 6
   and shared in section 7 — not just the happy path. Show the empty state and
   at least one notable non-default state for each page.
4. Keep the five-second test in mind: a first-time, inexperienced user must
   understand the screen with no training. If a screen fails that test,
   simplify it.

Constraints: dark theme first; calm surfaces with the red accent used only
where it carries meaning; flat and clean, no gradients or heavy shadows;
sentence case everywhere; one consistent component for each job across all
pages. Stay within the scope in section 9.

Before designing each page, briefly state your plan for it and any assumption
you are making where the brief is silent. Then design it. Move to the next
page only when the current one is complete.

Start with the Crew Cockpit.
